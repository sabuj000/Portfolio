Vision

Where I’m taking documentation next

A builder’s view of the problems coming for every documentation org — and the bets I’m making to get ahead of them. The common thread: documentation that serves people and AI equally, fixes itself from real signals, and proves its value in numbers.

Bet 1

🧩 Docs as the backbone of AI

AI assistants are becoming the main way developers consume documentation. I structure content to be AI-ready and LLM-friendly, so the same source serves people, search, and in-product AI assistants accurately — and so MCP-based agents and tools can rely on it.

Why now The cost of inaccurate or poorly structured docs multiplies once AI is reading them at scale.

Bet 2

📡 Signal-driven, proactive docs

The next frontier is documentation that fixes itself — support tickets, search queries, and usage signals feeding autonomous agents that close gaps before they generate support cost. I’ve already built the first version of this; the direction is making it continuous and trustworthy.

Why now Reactive, manual upkeep can’t scale with product velocity; signals can.

Bet 3

💹 Docs that prove their ROI

Documentation should defend its budget with numbers. I’m connecting content to fewer support tickets, feature adoption, and sales pipeline — so documentation reads as a measurable growth driver, not a cost center.

Why now Teams that can show impact get investment; teams that can’t get cut.

These aren’t abstractions — each builds directly on work I’ve shipped (AI agents, the docs CLI, analytics dashboards, and revenue-connected guidance). The vision is simply to push each one further and make it the default way documentation operates.
