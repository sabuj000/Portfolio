How I Work

Documentation, run like a product

I treat documentation as a product with its own strategy, standards, and lifecycle — and I run it the way engineering runs software: versioned, reviewed, automated, measured, and tied to business outcomes.

🛠️ Docs-as-code platform & tooling

I run documentation as code — versioned, reviewed, automated, and self-serve.

  • Built a documentation CLI bringing OpenAPI spec generation, content scaffolding, content sync, validation, and local preview into a single tool — enabling 79% of API doc PRs to come from non-docs engineers and nearly doubling the contributor base, without adding headcount.
  • Automated release notes and changelogs — cut release-notes effort from ~8 hours to ~15 minutes and made API docs the single source of truth, permanently closing a recurring broken-link gap. Extended the same auto-draft standard to more repositories.
  • Docs-as-code workflow — Git branches, pull requests, CI/CD validation, and content previews for every change, with reusable Markdown/MDX components.

🧭 Strategy & governance

  • Site structure & navigation — organize content so developers find the right answer in the fewest steps.
  • Content standards & style — drove 95% compliance with a defined English style standard across a large library, improving readability and consistency.
  • Editorial governance — define review gates, ownership, and freshness/audit cycles so docs stay accurate as the product changes.
  • Standards & document control — maintain compliance with ISO standards and document-control processes, and keep content easy to find and accessible.

🔍 Continuous content audits

I systematically review older and legacy documentation for accuracy and freshness — so people, LLMs, and MCP-based AI tools (search, in-product assistants, and agents) all get accurate answers during onboarding and act with precision on content they can trust.

📊 Data-informed documentation

I use data to validate assumptions, understand how docs are actually used, and feed insights back into the roadmap.

  • Built a single analytics dashboard (Looker Studio) monitoring 15+ content areas, with site-wide and page-level filters for views, engagement, and region.
  • Use GA4 and Pendo to understand discovery paths, engagement, and drop-offs — including separating internal vs. customer traffic for cleaner analytics.
  • Analyze session behavior with FullStory and log data with Splunk to spot confusion points and failure patterns.
  • Measure how AI bots and LLMs read and use documentation, keeping content accurate for AI-driven developer experiences.

💰 Tied to adoption & revenue

  • Built in-app banners and guidance that move users from docs into demos and sign-ups, connecting docs work to both self-serve and sales-led growth.
  • Connected product analytics with the CRM and outreach tools so in-app responses trigger automated follow-up — turning documentation into measurable sales pipeline.

🎥 Visual & video documentation

  • Run a sustained pipeline publishing 30+ how-to and conceptual videos every month, including an AI-assisted workflow for concept and value-prop videos.
  • Create workflow and system diagrams with Mermaid to explain API flows and developer journeys.
  • Use Camtasia and Trainn for short, task-focused videos, and collaborate with Product and Design in Figma.

👥 Scaled through contribution

  • PM-as-first-drafter model — PMs draft, the docs team reviews and publishes (128 PRs raised, 68 published), cutting documentation SLA by ~50%.
  • Contributor / buddy program — a broader internal and external network (60+ articles, 40+ PRs); when contributions get rejected, I look into why and fix the cause to improve quality and cut review time.

🤝 Cross-functional by default

I partner far beyond the docs team — Leadership, Product, Go-to-Market, Marketing, Support, Engineering, Design, and Developer Experience — aligning documentation with launches, campaigns, fewer support tickets, and the go-to-market plan.

🧠 My developer-experience perspective

I approach developer experience as a journey problem, not a documentation problem. I focus on:

  • Helping developers reach their first success quickly and experiment safely
  • APIs that are clear, predictable, and easy to recover from when something goes wrong
  • Complete, end-to-end workflows rather than isolated features
  • Practical examples, edge cases, and what to do when things fail
  • Content that works for people, SDKs, and AI tools alike
